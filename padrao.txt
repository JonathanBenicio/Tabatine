Analisando a estrutura do seu projeto **Tabatine** (um sistema construído com Next.js, integrando com Supabase e a API/Webhooks do ERP Omie), você já possui uma base bem estruturada com separação de *components*, *hooks*, *store* e *api*. 

Para elevar a maturidade do projeto focando em **legibilidade, organização, desempenho, observabilidade, testes e sustentabilidade**, aqui estão os padrões e boas práticas recomendados, divididos por área:

---

### 1. Organização e Legibilidade (Arquitetura)

* **Separação de Responsabilidades (Controller-Service-Repository):**
    * Suas rotas de API (`src/app/api/...`) não devem conter regras de negócio complexas. Elas devem atuar apenas como *Controllers* (receber a requisição, validar e retornar a resposta).
    * Crie uma camada de **Services** (ex: `src/services/omieSyncService.ts`) para conter a lógica de negócio (ex: "buscar no Omie e salvar no Supabase").
    * Mantenha a interação com o banco de dados em um padrão **Repository** ou em utilitários bem definidos em `src/utils/supabase/`.
* **Domain-Driven Design (DDD) "Light" / Feature-Sliced Design:**
    * Conforme o projeto cresce, agrupar arquivos por *tipo* (`/hooks`, `/components`, `/store`) pode ficar confuso. Considere agrupar por **Domínio** (ex: criar uma pasta `src/features/vendas` que contém seus próprios componentes, hooks e stores locais), deixando na raiz de `components` apenas os genéricos (botões, modais).
* **Padrão de Nomenclatura Estrito:**
    * Mantenha a consistência. Notei que há misturas como `useNfQuery.ts` e `useContasCorrentesQuery.ts` (camelCase) com pastas de rotas como `nf` e `contas-correntes` (kebab-case). Padronize para facilitar a busca.

### 2. Desempenho (Performance)

* **Server Components vs. Client Components (Next.js App Router):**
    * Identifique componentes de UI pesados e mantenha a maior parte da aplicação como Server Components (sem a diretiva `"use client"`). 
    * Use `"use client"` apenas onde há interatividade real (ex: `onClick`, `useState`, ou uso do Zustand no `src/store/`). Isso reduz muito o JavaScript enviado ao navegador.
* **Estratégias de Cache no Next.js e React Query:**
    * Aproveite o cache de *fetch* nativo do Next.js para dados estáticos ou que mudam pouco (ex: listar formas de pagamento, características de produtos).
    * No lado do cliente, já que está usando *hooks* (`useVendasQuery.ts`), se estiver usando **TanStack React Query**, certifique-se de configurar o `staleTime` corretamente para evitar *re-fetches* desnecessários ao focar a aba do navegador.
* **Virtualização de Tabelas Grandes:**
    * Componentes como `VendasTable.tsx` e `ClientesTable.tsx` podem sofrer engasgos ao renderizar milhares de linhas. Considere usar bibliotecas como `@tanstack/react-virtual` para renderizar apenas os itens visíveis na tela.

### 3. Sustentabilidade e Resiliência (Lidando com o Omie e Supabase)

* **Tratamento de Erros e Rate Limiting (Padrão Circuit Breaker / Retries):**
    * APIs de ERPs como o Omie costumam ter limites de requisição (*rate limits*) e podem apresentar instabilidades temporárias. Implemente um padrão de repetição (*Exponential Backoff Retry*) ao buscar dados na API.
* **Idempotência em Webhooks:**
    * No arquivo `src/app/api/webhooks/omie/route.ts`, garanta que seu processamento seja **idempotente**. O Omie pode enviar o mesmo webhook de "Venda Atualizada" duas vezes por falha de rede. Seu sistema não pode duplicar os registros no Supabase caso receba a mesma notificação duas vezes.
* **Validação de Dados (Zod):**
    * Nunca confie cegamente no payload que vem do cliente ou da API externa. Use bibliotecas como **Zod** para criar *schemas* e validar tanto os formulários no frontend quanto as requisições que chegam nas suas rotas (`route.ts`).

### 4. Observabilidade

* **Logging Estruturado:**
    * Substitua os tradicionais `console.log` por uma biblioteca de log estruturado (como o `Pino` ou `Winston`). Registre eventos em formato JSON, o que facilita buscas em ferramentas de monitoramento.
    * **O que logar?** Início e fim de sincronizações com o Omie, falhas de autenticação, tempos de resposta de consultas pesadas no Supabase e payloads inválidos.
* **Application Performance Monitoring (APM) e Error Tracking:**
    * Integre ferramentas como **Sentry** ou **Datadog**. Quando um cliente tiver um erro não tratado no frontend ou quando um Webhook falhar no backend, você receberá a trilha de erro (*stack trace*) imediatamente, antes mesmo de o cliente reclamar.

### 5. Testes

Você já possui arquivos de teste como `src/lib/ofxParser.test.ts` e `src/utils/supabase/filter-utils.test.ts`. Isso é excelente! Para expandir:

* **Testes Unitários:**
    * Foque em testar a lógica pura: os parsers (como você já faz), os métodos de filtragem, e a lógica dentro das suas stores do Zustand.
* **Testes de Integração (Mocking APIs externas):**
    * Teste as rotas da sua API usando ferramentas como *MSW (Mock Service Worker)* para simular as respostas da API do Omie e verificar se seu banco de dados (Supabase) seria atualizado corretamente, sem fazer requisições reais durante a CI/CD.
* **Testes End-to-End (E2E):**
    * Para os fluxos mais críticos, como "Fazer login" ou "Verificar conciliação bancária" (`src/app/conciliacao/page.tsx`), escreva alguns testes usando **Playwright** ou **Cypress** para garantir que a interface responde perfeitamente de ponta a ponta.

**Um ótimo primeiro passo prático:**
Comece revisando a rota de `webhooks` do Omie para adicionar logs estruturados e garantir idempotência, pois essa é a porta de entrada de dados automáticos do seu sistema. Em seguida, blinde suas respostas da API usando *Zod* para garantir contratos estritos entre backend e frontend.
